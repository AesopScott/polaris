# Registries

**Registries** document cross-boundary contracts in Polaris — places where two or more parts of the codebase refer to the same name independently and must agree.

## What is a boundary?

A boundary exists wherever:
- Client code sends a message that server code receives
- Code writes to persistent storage that other code reads
- Code sets a value (environment variable, token claim, API parameter) that other code checks
- Any two independent code paths reference the same name and can disagree

Mismatches across boundaries are the hardest bugs to catch — the code is correct in isolation, but the integration breaks.

## How to use these registries

**When adding a new boundary name (new WebSocket event, new API endpoint, new database collection, etc.):**
1. Make your change in code
2. Update the relevant registry file to add the new name and its producers/consumers
3. Include the registry update in your commit

**When changing an existing boundary (adding a field to a WebSocket event, renaming an endpoint, etc.):**
1. Make your change in code
2. Update the relevant registry to reflect the change
3. Include the registry update in the same commit

**When unsure if something is a boundary:**
- If the name appears in two or more files, it's probably a boundary
- If the name could be refactored in one file without breaking others (e.g., a local variable), it's not a boundary
- When in doubt, ask: "Can this name be spelled differently in different places?"

## Running the audit

To verify all boundaries in the codebase match their registry definitions:

```
/cross-boundary-audit
```

This command:
1. Scans the codebase for all references to boundary names
2. Compares them against the registry definitions
3. Reports any gaps (producers without consumers, shape mismatches, etc.)
4. Updates the Audit Trail section in each registry

The audit is read-only — it does not modify code, only registries (to update their audit trail).
